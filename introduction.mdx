## What mentalmodel is for

`mentalmodel` is for programs where architecture matters after the first week.

Instead of relying on ad hoc orchestration code, conventions, and tribal
knowledge, you author against a small set of semantic primitives. That authored
structure becomes the source of truth for:

- static analysis
- Mermaid diagrams
- generated docs
- runtime execution
- semantic execution records
- `.runs` debugging bundles
- OpenTelemetry traces and metrics

The package is not trying to replace Python. It is trying to make workflow
structure, side effects, state transitions, and correctness checks explicit
enough that both humans and coding agents can keep the right mental model of
the program.

## The core idea

Most workflow-like systems get harder to reason about over time because the
important semantics are implicit:

- which code is stateful
- which code performs side effects
- which nodes depend on which other nodes
- what invariants must hold
- where code is allowed to run

`mentalmodel` makes those decisions explicit in code.

You describe a program using primitives like:

- `Workflow`
- `Actor`
- `Effect`
- `Invariant`
- `Parallel`
- `Join`
- `Ref`

And when you need an extension, you add one through the plugin system, as with
`RuntimeContext` and `AutoResearch`.

## What the package already does well

### One authored model powers multiple surfaces

The same workflow definition can drive:

- `mentalmodel check`
- `mentalmodel graph`
- `mentalmodel docs`
- `mentalmodel verify`
- `.runs` inspection
- replay and diffing

That means diagrams and debugging artifacts stay aligned with the code instead
of drifting.

### Semantic runtime records come first

OpenTelemetry is supported, but it is not the package's source of truth.

The primary runtime ledger is the semantic event stream recorded by the
executor. That is what powers:

- `runs trace`
- `runs inputs`
- `runs outputs`
- `replay`
- run diffing

This is an important distinction: `mentalmodel` is designed so that you can
still understand a run even without a hosted observability backend.

### The extension model is real

Plugins are not just documentation hooks. `mentalmodel` already supports
runtime-executable plugins. That is how the `AutoResearch` demo works today.

## What it is not

`mentalmodel` is not:

- a domain-specific RL package
- a generic business-workflow engine like Temporal
- a replacement for Python application structure
- a hidden magic runtime that infers everything from ordinary code

It is a programming model with explicit semantics.

## Current core primitives

| Primitive | Purpose |
| --- | --- |
| `Workflow` | Names and groups a program |
| `Actor` | Stateful semantic node |
| `Effect` | Explicit side-effect boundary |
| `Invariant` | Runtime correctness check |
| `Parallel` | Fanout grouping |
| `Join` | Explicit synchronization / merge point |
| `Ref` | Data dependency edge |
| `RuntimeContext` | Plugin-based runtime boundary annotation |

## A better way to think about it

Use `mentalmodel` when you want the answer to questions like these to be easy:

- What can run concurrently?
- Which node produced this value?
- What side effect just happened?
- Which invariant failed, with which exact inputs?
- What changed between two runs?
- Which runtime boundary did this node execute under?

If those questions matter, the package is a good fit.
