---
title: mentalmodel remote
description: Bootstrap and operate the first remote-runs localhost stack.
---

`mentalmodel remote` is the first operator-facing command for the remote runs
data plane.

## Commands

```bash
uv run mentalmodel remote link
uv run mentalmodel remote status
uv run mentalmodel remote publish-catalog
uv run mentalmodel remote doctor
uv run mentalmodel remote write-demo --output-dir /tmp/mentalmodel-remote
uv run mentalmodel remote doctor --workspace-config /tmp/mentalmodel-remote/workspace.toml
uv run mentalmodel remote up --output-dir /tmp/mentalmodel-remote --open-browser
uv run mentalmodel remote sync --server-url http://127.0.0.1:8765
```

## Choosing a mode

`mentalmodel remote` now has two explicit modes:

- hosted repo-linked mode
- local workspace mode

Hosted repo-linked mode is the primary product path.

Local workspace mode exists for generated demo stacks, offline development, and
multi-repo local testing.

## Repo-linked hosted path

Phase 1 introduces a repo-owned service config in `mentalmodel.toml`.

Illustrative shape:

```toml
[project]
project_id = "pangramanizer"
label = "Pangramanizer"

[remote]
server_url = "https://mentalmodel.example.com"
api_key_env = "MENTALMODEL_API_KEY"
default_environment = "prod"

[catalog]
provider = "pangramanizer.mentalmodel_training.dashboard:catalog"
publish_on_link = true

[runs]
default_runs_dir = ".runs"

[verify]
default_spec = "pangramanizer/mentalmodel_training/verification/real_smoke.toml"
```

The common hosted setup flow is now:

```bash
uv run mentalmodel remote link
uv run mentalmodel remote status
uv run mentalmodel remote publish-catalog
```

`remote link` discovers `mentalmodel.toml`, resolves the repo-owned catalog
provider, registers the project remotely, and publishes the current catalog
snapshot. `remote status` asks the remote service for the linked project record
so you can confirm:

- which project is linked
- which server it is linked to
- whether a catalog snapshot is published
- how many catalog entries are currently stored remotely
- which completed run was uploaded most recently

This is intentionally separate from the local `workspace.toml` path. Workspace
files are for local stack work, not the standard hosted setup path. The hosted
direction is now repo-owned config plus remote project records.

## `remote link`

Link the current repo-owned project to a remote service:

```bash
uv run mentalmodel remote link
uv run mentalmodel remote link --config /Users/ben/repos/pangramanizer/mentalmodel.toml
```

What it does:

- discovers or reads `mentalmodel.toml`
- validates the repo-owned config contract
- resolves the configured API token from the configured env var
- loads the Python catalog provider and serializes a catalog snapshot
- `POST`s the linked project record to the remote service

How to use that:

1. Add `mentalmodel.toml` to the repo root.
2. Set the configured API key environment variable.
3. Run `mentalmodel remote link`.
4. Open the hosted dashboard or use `mentalmodel remote status` to confirm the
   project is now known remotely.

## `remote publish-catalog`

Republish just the hosted dashboard contract for an already linked project:

```bash
uv run mentalmodel remote publish-catalog
uv run mentalmodel remote publish-catalog --config /Users/ben/repos/pangramanizer/mentalmodel.toml
```

Use this when:

- pinned nodes changed
- metric groups changed
- custom views changed
- the hosted dashboard should pick up catalog updates without relinking the
  whole project

`remote publish-catalog` reads the same repo-owned `mentalmodel.toml`, resolves
the configured Python catalog provider, serializes the current dashboard
contract into a stable snapshot, and `POST`s that snapshot to the remote
service.

The hosted dashboard now reads those stored remote catalog snapshots first. In
the hosted path it no longer needs to import the producer repo locally just to
render project entries, metric groups, pinned nodes, or custom views.

## `remote status`

Read the linked project record back from the remote service:

```bash
uv run mentalmodel remote status
uv run mentalmodel remote status --config /Users/ben/repos/pangramanizer/mentalmodel.toml
```

Use this when you want to confirm the Phase 1 hosted seam before thinking about
run upload or live streaming.

The status payload now also surfaces catalog version and publish metadata, so
you can confirm that the hosted dashboard is looking at the expected snapshot.
In Phase 3 it also surfaces the last successful completed-run upload, so you
can verify that normal `mentalmodel verify` executions are arriving remotely.

## `remote doctor`

Validate either hosted repo-linked mode or local workspace mode:

```bash
uv run mentalmodel remote doctor
uv run mentalmodel remote doctor --config /Users/ben/repos/pangramanizer/mentalmodel.toml
uv run mentalmodel remote doctor --workspace-config /tmp/mentalmodel-remote/workspace.toml
```

Mode selection rules:

- if you pass `--workspace-config`, `remote doctor` validates local workspace mode
- otherwise it validates hosted repo-linked mode
- in hosted mode it auto-discovers `mentalmodel.toml` from the current repo if you do not pass `--config`

Hosted checks now tell you whether:

- the repo-owned config loads
- the API key resolves
- the catalog snapshot can be built
- the default verify spec and runs dir are usable
- the remote project record is reachable

Local checks still validate:

- the generated workspace registry
- provider resolution
- output routing
- stack asset layout

## Normal completed-run flow

In a linked repo, the normal operator path is now just:

```bash
uv run mentalmodel verify --spec path/to/spec.toml
```

When `mentalmodel.toml` is present and linked:

- the workflow still writes the canonical local `.runs` bundle
- after local bundle materialization succeeds, the completed run uploads to the
  remote service automatically
- `mentalmodel verify --json` includes `runtime.completed_run_upload` so you can
  correlate the local bundle path with the remote run record
- `mentalmodel remote status` reflects the most recent successful completed-run
  upload

If remote upload fails, the local `.runs` bundle is still intact and usable for
inspection or replay. The command surfaces the upload failure explicitly so you
can recover without guessing whether the local run was preserved.

## Live execution flow

In Phase 4, the same linked `mentalmodel verify --spec ...` path also streams
live execution state to the remote service while the workflow is still running.

That means the hosted dashboard can show:

- a run row before the completed bundle exists
- a live graph snapshot for the current run
- live semantic records as they arrive
- live spans
- early numeric metrics derived from streamed node outputs
- early invariant status derived from streamed records

The important semantics are:

- local `.runs` is still the canonical completed artifact
- live streaming is best-effort and must never break local execution
- the completed bundle upload is the authoritative closeout step
- once the completed bundle lands, the dashboard should pivot back to the
  persisted run view rather than treating the live session as canonical

What that looks like operationally:

1. `mentalmodel verify --spec ...` starts.
2. The producer opens a remote live session for the linked project.
3. Records and spans stream to the remote service in batches.
4. The hosted dashboard lists the run immediately as a live run.
5. Local `.runs` materialization completes.
6. The completed bundle uploads.
7. The remote live session is marked committed and the dashboard now reads the
   persisted run as the source of truth.

If live upload fails in the middle of execution, the workflow still keeps
running locally and the final `.runs` bundle is still written. If the completed
bundle upload succeeds, the hosted dashboard still receives the final canonical
run even if some live updates were missed.

## `remote write-demo`

Write a local remote-runs demo directory:

```bash
uv run mentalmodel remote write-demo \
  --output-dir /tmp/mentalmodel-remote

uv run mentalmodel remote write-demo \
  --output-dir /tmp/mentalmodel-remote \
  --mentalmodel-root /Users/ben/repos/mentalmodel \
  --pangramanizer-root /Users/ben/repos/pangramanizer
```

The generated directory includes:

- `docker-compose.remote-minimal.yml` for Postgres + MinIO
- `mentalmodel.remote.env` with wired local credentials
- `workspace.toml` with registered projects
- `run-dashboard.sh` to launch the shared dashboard/API
- `start-stack.sh` to start the backend services and launch the dashboard
- `stop-stack.sh` to stop the backend services
- `sync-local-runs.sh` to upload existing local `.runs`
- `otel/` with the existing LGTM traces demo
- `REMOTE-DEMO.md` with the local operator flow

## `remote doctor`

Validate the generated workspace, catalog providers, and writable run storage:

```bash
uv run mentalmodel remote doctor \
  --workspace-config /tmp/mentalmodel-remote/workspace.toml
```

`remote doctor --workspace-config ...` now also fails if an enabled project has no explicit `runs_dir`.
That is a real launch-routing error for the shared stack, not just missing
metadata.

## `remote up`

Start the generated backend services and then launch the dashboard in one step:

```bash
uv run mentalmodel remote up \
  --output-dir /tmp/mentalmodel-remote \
  --open-browser
```

This is the quickest Phase 3 local-stack path. It runs:

- `docker compose -f docker-compose.remote-minimal.yml up -d`
- `mentalmodel ui` with the generated workspace and `MENTALMODEL_REMOTE_*`
  backend configuration

Stop the backend services later with:

```bash
uv run mentalmodel remote down --output-dir /tmp/mentalmodel-remote
```

## `remote sync`

Sync one or more persisted local runs into the remote ingest API:

```bash
uv run mentalmodel remote sync \
  --runs-dir /tmp/mentalmodel-runs

uv run mentalmodel remote sync \
  --config /Users/ben/repos/pangramanizer/mentalmodel.toml \
  --graph-id async_rl_demo \
  --run-id run-123

uv run mentalmodel remote sync \
  --server-url http://127.0.0.1:8765 \
  --graph-id async_rl_demo \
  --run-id run-123

uv run mentalmodel remote sync \
  --server-url http://127.0.0.1:8765 \
  --project-id pangramanizer-training \
  --project-label "Pangramanizer Training" \
  --environment-name localhost \
  --catalog-source module-provider
```

## What it does

- resolves one or more persisted local run bundles from `.runs`
- builds a canonical `RunManifest` plus artifact descriptors for each run
- uploads each run bundle to `POST /api/remote/runs`
- returns the synced remote upload receipts in table or JSON form

The current remote backend is intentionally minimal and deterministic. The API
stores run manifests in Postgres and artifact blobs in S3-compatible object
storage. The dashboard read path materializes remote bundles into a local cache
so the existing historical inspection APIs can keep using the current run
inspection helpers.

Treat `remote sync` as the manual recovery and backfill path now, not the
normal hosted workflow. Use it when:

- a linked `verify` run uploaded locally but the remote publish step failed
- you want to backfill older local runs into the hosted service
- you are operating against a local stack that is not using the repo-linked
  automatic path

For local development and deterministic tests, the older file-backed store still
exists as a fallback. It is no longer the intended Phase 2 backend.

## How to use that

1. Write the localhost stack assets:

```bash
uv run mentalmodel remote write-demo --output-dir /tmp/mentalmodel-remote
```

2. Start the shared stack:

```bash
uv run mentalmodel remote up \
  --output-dir /tmp/mentalmodel-remote \
  --open-browser
```

3. Materialize one or more local runs:

```bash
uv run mentalmodel verify \
  --spec /Users/ben/repos/mentalmodel/src/mentalmodel/examples/review_workflow/review_workflow_fixture.toml \
  --runs-dir /tmp/local-runs
```

4. Sync the local run into that stack manually if you are testing the recovery
   path:

```bash
uv run mentalmodel remote sync \
  --server-url http://127.0.0.1:8765 \
  --runs-dir /tmp/local-runs \
  --project-id mentalmodel-examples \
  --project-label "Mentalmodel Examples"
```

5. Inspect the synced run through the existing dashboard or `/api/runs` APIs.

For project-routed launches, `mentalmodel verify` and dashboard launches should
write directly into the shared stack without requiring `remote sync`.

For the built-in dashboard proof path, use the `review_workflow_fixture.toml`
spec above. The default built-in catalog is scoped to the review workflow
entries, so syncing an unrelated graph such as `async_rl_demo` will not appear
under the selected built-in catalog entry.

## Running the UI Against The Durable Backend

`mentalmodel ui` can point its historical read and ingest paths at the Phase 2
remote backend directly:

```bash
uv run mentalmodel ui \
  --remote-database-url postgresql://postgres:postgres@127.0.0.1:5432/mentalmodel \
  --remote-object-store-bucket mentalmodel-runs \
  --remote-object-store-endpoint http://127.0.0.1:9000 \
  --remote-object-store-access-key minio \
  --remote-object-store-secret-key miniosecret \
  --no-remote-object-store-secure
```

Those flags also respect the matching `MENTALMODEL_REMOTE_*` environment
variables, which is the intended handoff point for the later self-hosted Phase
3 stack templates.

## Notes

- `remote write-demo` is the current localhost bootstrap path for the remote MVP.
- `--run-id` targets one specific run; omitting it syncs all matching runs.
- `--graph-id` and `--invocation-name` narrow which local runs are selected.
- project and catalog flags let one shared stack distinguish runs from multiple
  sources such as `mentalmodel` demos and Pangramanizer training flows.

## UI Acceptance

Use the browser-backed acceptance harness when you want to prove that a UI
launch actually turns into a historical run:

```bash
python /Users/ben/repos/mentalmodel/scripts/remote_ui_acceptance.py \
  --server-url http://127.0.0.1:8765 \
  --spec-id pangramanizer-training:real-smoke \
  --project-id pangramanizer-training \
  --graph-id pangramanizer_training_real_smoke \
  --invocation-name pangram_real_smoke
```

The script uses `agent-browser` to launch from the dashboard UI, then polls
`/api/runs` until it observes a new run or times out with a screenshot and JSON
diagnostics.
