---
title: mentalmodel ui
description: Launch the hosted dashboard UI on top of persisted .runs data and live verification sessions.
---

`mentalmodel ui` starts the FastAPI-backed dashboard surface for local and
hosted-style inspection workflows.

## Command

```bash
uv run mentalmodel ui
```

Common flags:

```bash
uv run mentalmodel ui --host 0.0.0.0 --port 8765
uv run mentalmodel ui --runs-dir /tmp/mentalmodel-runs
uv run mentalmodel ui --workspace-config /tmp/mentalmodel-remote/workspace.toml
uv run mentalmodel ui --frontend-dist apps/dashboard/dist --open-browser
uv run mentalmodel ui --frontend-dev-url http://127.0.0.1:5173
uv run mentalmodel ui --catalog-entrypoint mypkg.dashboard:catalog
uv run mentalmodel ui --remote-database-url postgresql://... --remote-object-store-bucket mentalmodel-runs
```

Mode guidance:

- hosted mode: use repo-linked projects plus the remote backend
- local mode: use `--workspace-config` and optional `--catalog-entrypoint`

## What it serves

- a React dashboard if `apps/dashboard/dist` exists
- a Vite frontend in local dev mode when you point the browser at
  `--frontend-dev-url`
- JSON APIs for:
  - launchable workflow specs
  - live execution sessions
  - persisted run summaries
  - persisted run graphs
  - replay views
  - node inputs, outputs, and traces

## How to use that

## Hosted mode

Hosted mode is the primary product path. The dashboard should render linked
projects from remote project records and stored catalog snapshots without
importing producer repos locally.

Typical hosted flow:

1. Link the repo:

```bash
uv run mentalmodel remote link
uv run mentalmodel remote status
```

2. Launch the UI against the remote backend:

```bash
uv run mentalmodel ui \
  --remote-database-url postgresql://postgres:postgres@127.0.0.1:5432/mentalmodel \
  --remote-object-store-bucket mentalmodel-runs \
  --remote-object-store-endpoint http://127.0.0.1:9000 \
  --remote-object-store-access-key minio \
  --remote-object-store-secret-key miniosecret \
  --no-remote-object-store-secure
```

Do not use `--workspace-config` or `--catalog-entrypoint` for the normal hosted
path.

## Local mode

For the built-in proof path:

1. Build the frontend once:

```bash
cd apps/dashboard
npm install
npm run build
```

2. Start the UI:

```bash
cd /path/to/mentalmodel
uv run mentalmodel ui --open-browser
```

For a rebuild-free local dev loop:

```bash
cd apps/dashboard
npm install
npm run dev:stack
```

That uses Vite on `http://127.0.0.1:5173` and points it at the Python API
running on `http://127.0.0.1:8765`.

3. In the dashboard:

- choose `Ticket Review Fixture` or `Ticket Review Strict`
- run the workflow
- inspect the graph panel to drill into nodes
- click grouped metrics to jump directly to the node and frame that emitted them
- inspect the invariant list to jump directly to constraint nodes
- inspect the records table for the full logs-like semantic stream

## Custom catalogs

The dashboard is catalog-driven. A provider entrypoint can return either:

- one or more `DashboardCatalogEntry` values
- one `ProjectCatalog` value for a project-scoped catalog

Run a custom catalog with:

```bash
uv run mentalmodel ui \
  --catalog-entrypoint mypkg.dashboard:catalog \
  --frontend-dev-url http://127.0.0.1:5173
```

That lets downstream projects like Pangramanizer expose their own specs, metric
groups, pinned nodes, and default loop drilldown behavior without forking the
dashboard shell during local development.

Project-scoped catalogs are the preferred shape when one shared stack needs to
host multiple projects cleanly.

## Workspace configs

`--workspace-config` is the preferred way to run one shared local stack across
multiple repos.

The workspace TOML registers projects, their repo roots, and their provider
entrypoints. `mentalmodel ui` resolves each enabled provider, including
providers that live in sibling repos such as Pangramanizer, and exposes the
combined catalog from one dashboard instance.

Example:

```bash
uv run mentalmodel ui \
  --runs-dir /tmp/mentalmodel-remote/data \
  --workspace-config /tmp/mentalmodel-remote/workspace.toml \
  --open-browser
```

For the generated localhost stack, use:

```bash
uv run mentalmodel remote write-demo --output-dir /tmp/mentalmodel-remote
uv run mentalmodel remote up --output-dir /tmp/mentalmodel-remote --open-browser
```

## Remote backend flags

For the hosted remote backend path, point the UI at Postgres plus an
S3-compatible object store:

```bash
uv run mentalmodel ui \
  --remote-database-url postgresql://postgres:postgres@127.0.0.1:5432/mentalmodel \
  --remote-object-store-bucket mentalmodel-runs \
  --remote-object-store-endpoint http://127.0.0.1:9000 \
  --remote-object-store-access-key minio \
  --remote-object-store-secret-key miniosecret \
  --no-remote-object-store-secure
```

The same values can be supplied through `MENTALMODEL_REMOTE_*` environment
variables.

## What to look for

The review workflow proof case should show:

- different resolution metrics for fixture vs strict runs
- loop frame ids like `ticket_review_loop[0]`
- warning-severity invariant results attached to the loop body
- root `queue_summary` outputs with `processed`, `escalations`, and `auto_publish`

For a linked hosted project, the same dashboard should also show the live to
completed handoff clearly:

- a run can appear in the list before the final bundle exists
- records, spans, metrics, and invariant status update from the live session
- once the completed bundle is committed, the run detail pivots back to the
  persisted `.runs` artifact as the canonical inspection surface
- if live streaming drops temporarily, the final completed bundle still
  preserves the authoritative run

Phase 6 also adds explicit remote delivery health diagnostics to the overview
surface:

- remote delivery health for the selected project or run
- the producer-side `runtime.live_execution_delivery` result for active runs
- recent remote operation events for link, catalog publish, live delivery, and
  completed run upload activity
