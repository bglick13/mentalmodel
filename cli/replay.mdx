---
title: "replay"
description: "Reconstruct the full semantic event timeline for one persisted run."
---

The `replay` command reads a persisted run bundle from disk and reconstructs a chronological semantic event timeline. It shows each event in sequence order along with a per-node summary of invariant outcomes.

## Usage

```bash
mentalmodel replay --graph-id <graph_id>
```

## Flags

<ParamField query="--graph-id" type="string" required>
  The graph ID to replay. Must match a directory under the runs root (e.g. `.runs/<graph_id>/`).
</ParamField>

<ParamField query="--run-id" type="string">
  The specific run ID to replay. When omitted, the newest run for the graph is used.
</ParamField>

<ParamField query="--frame-id" type="string">
  Restrict replay to one execution frame, such as `steps[3]`.
</ParamField>

<ParamField query="--loop-node-id" type="string">
  Restrict replay to frames owned by one loop node.
</ParamField>

<ParamField query="--iteration-index" type="integer">
  Restrict replay to one loop iteration.
</ParamField>

<ParamField query="--runs-dir" type="path">
  Root directory for persisted run artifacts. Defaults to `./.runs`.
</ParamField>

<ParamField query="--json" type="flag">
  Emit a single JSON object to stdout instead of rendering Rich tables.
</ParamField>

## Output

When you run without `--json`, the command renders three tables.

**Summary table:**

| Field | Value |
|---|---|
| Graph | my-graph |
| Run | run-abc123 |
| Schema | 2 |
| Success | yes |
| Verification | yes |
| Frames | 1 |
| Events | 14 |
| Nodes | 4 |
| Runtime Error | |

**Replay Events table** — one row per semantic event in sequence order:

| Seq | Frame | Node | Event | Timestamp | Payload |
|---|---|---|---|---|---|
| 1 | root | collect_data | node.started | 1712345678000 | `{"input_count": 0, "kind": "actor"}` |
| 2 | root | collect_data | node.succeeded | 1712345678042 | `{"kind": "actor", "output_type": "dict"}` |
| 3 | root | process_results | node.started | 1712345678043 | `{"input_count": 1, "kind": "effect"}` |

**Replay Node Summary table** — one row per node:

| Node | Frame | Iteration | Events | First Seq | Last Seq | Last Event | Invariant |
|---|---|---|---|---|---|---|---|
| collect_data | root | | 2 | 1 | 2 | node.succeeded | pass |
| process_results | root | | 2 | 3 | 4 | node.succeeded | |

If an invariant fails with `severity="warning"`, the node summary shows `warning_fail`. Fatal invariant failures show `error_fail`.

## How to interpret replay

Use `replay` when you want a whole-run narrative instead of one-node detail.

When loops arrive, `replay` becomes the fastest way to answer two related
questions:

- what happened in the whole run?
- what happened inside one specific iteration or frame?

The most useful reading pattern is:

1. skim the summary table
2. scan the event table for where the run changes phase or fails
3. use the node summary table to identify nodes with unusual event counts or
   failed invariants
4. jump from there to `runs trace`, `runs inputs`, or `runs outputs`

`replay` is especially useful for answering:

- what happened first?
- which node failure cascaded into the rest of the run?
- did the run order match my mental model of the workflow?

## Examples

<CodeGroup>
```bash Replay latest run
mentalmodel replay --graph-id my-graph
```

```bash Replay a specific run
mentalmodel replay --graph-id my-graph --run-id run-abc123
```

```bash JSON output
mentalmodel replay --graph-id my-graph --json
```

```bash Replay one iteration
mentalmodel replay --graph-id my-graph --frame-id steps[3]
```

```bash Custom runs directory
mentalmodel replay \
  --graph-id my-graph \
  --runs-dir /tmp/my-runs
```
</CodeGroup>

## Replay vs. runs show

`replay` and [`runs show`](/cli/runs) serve different purposes:

- **`replay`** reconstructs the full ordered event timeline. It reads `records.jsonl` and renders events in sequence order with timestamps and payloads. Use it when you want to understand _what happened_ and _in what order_ during a run.
- **`runs show`** shows the static metadata and file index for a bundle — graph ID, schema version, record counts, trace configuration, and paths to each file. Use it when you want a quick summary of a run or need to locate the raw bundle files.

The two commands pair well:

1. `runs show` to find the right run
2. `replay` to understand the whole run
3. `runs trace` for a specific node
