---
title: "verify"
description: "Run static analysis, execute the workflow, run property checks, and persist a `.runs` bundle."
---

`mentalmodel verify` is the main end-to-end validation command.

It combines:

1. static analysis
2. real runtime execution
3. property-check execution when the module defines any

and writes a run bundle to `.runs/`.

## Usage

```bash
mentalmodel verify --entrypoint module:function
```

It can now also bind a separate runtime environment and read a full invocation
spec from TOML:

```bash
mentalmodel verify \
  --entrypoint mypkg.workflow:build_program \
  --environment-entrypoint mypkg.runtime:build_environment
```

```bash
mentalmodel verify --spec verification/smoke.toml
```

## Flags

<ParamField query="--entrypoint" type="string" default="mentalmodel.examples.async_rl.demo:build_program">
  Workflow entrypoint in `module.submodule:function_name` format.
</ParamField>

<ParamField query="--runs-dir" type="path">
  Root directory for persisted run artifacts. Defaults to `./.runs`.
</ParamField>

<ParamField query="--params-json" type="string">
  Optional JSON object passed as keyword arguments to a callable entrypoint.
</ParamField>

<ParamField query="--params-file" type="path">
  Path to a JSON file passed as keyword arguments to a callable entrypoint.
</ParamField>

<ParamField query="--environment-entrypoint" type="string">
  Optional runtime environment entrypoint in `module.submodule:function_name`
  format.
</ParamField>

<ParamField query="--environment-params-json" type="string">
  Optional JSON object passed as keyword arguments to a callable environment
  entrypoint.
</ParamField>

<ParamField query="--environment-params-file" type="path">
  Path to a JSON file passed as keyword arguments to a callable environment
  entrypoint.
</ParamField>

<ParamField query="--invocation-name" type="string">
  Optional run-level label persisted in `.runs` and attached to emitted
  telemetry. Good examples are `real_smoke`, `shadow_verify1`, or
  `training_prod`.
</ParamField>

<ParamField query="--spec" type="path">
  Optional TOML verify spec describing the workflow factory, runtime
  environment factory, and invocation metadata.
</ParamField>

<ParamField query="--json" type="flag">
  Emit structured JSON instead of Rich tables.
</ParamField>

Use only one of `--params-json` or `--params-file`, and only one of
`--environment-params-json` or `--environment-params-file`.

## What verify actually does

### 1. Static analysis

This is the same analysis used by `mentalmodel check`.

It catches current graph-level issues such as:

- duplicate node ids
- malformed edge references
- missing workflow roots
- missing runtime-context metadata
- missing plugin provenance

### 2. Runtime execution

The workflow is executed by the async runtime.

This produces:

- node outputs
- state transitions
- semantic execution records
- local span capture when enabled

### 3. Property checks

If the entrypoint module defines registered property checks, `verify` runs them
after runtime execution.

This is how demo workflows can combine:

- runtime invariants inside the graph
- broader Hypothesis-backed checks outside the graph

## What gets written

A successful or failed verify run writes:

```text
.runs/<graph_id>/<run_id>/
```

Typically containing:

- `summary.json`
- `verification.json`
- `records.jsonl`
- `outputs.json`
- `state.json`
- `otel-spans.jsonl`

Use:

```bash
mentalmodel runs latest --graph-id <graph_id>
mentalmodel runs trace --graph-id <graph_id> --node-id <node_id>
```

immediately after `verify` when debugging.

## Reading the output

Without `--json`, the command renders:

- a summary table
- a runtime verification table
- a property-check table when checks exist

The most important fields are:

- static errors
- warnings
- runtime success/failure
- warning invariant failures
- invocation name
- run artifact directory
- property check success/failure

Example:

```text
mentalmodel verify summary
Graph       Static Errors  Warnings  Runtime  Property Checks
async_rl_demo 0            0         pass     2

Runtime Verification
Success  Records  Outputs  State Entries  Invocation       Warning Invariants  Run Artifacts
yes      48       10       3              real_smoke       1                   .runs/async_rl_demo/run-abc123
```

How to use that:

- if `Static Errors` is nonzero, fix the graph before trusting runtime output
- if `Runtime` failed, go straight to the run bundle and inspect `runs trace`
- if `Warning Invariants` is nonzero, the run continued but you still have advisory correctness drift to inspect
- if `Invocation` is set, use it later with `runs list --invocation-name ...`
  to separate smoke runs from training runs
- if runtime passed but a property check failed, inspect the property-check name
  first, then compare it to runtime outputs and records
- always note the `Run Artifacts` path; that is the anchor for deeper debugging

## Parameter validation

Parameterized verification is intentionally strict:

- the payload must decode to a JSON object
- the entrypoint must resolve to a callable, not a prebuilt `Workflow`
- keys must match the callable signature

That means this works well for real workflow factories:

```python
def build_program(*, config_path: str, mode: str) -> Workflow[NamedPrimitive]:
    ...
```

and fails fast for malformed payloads instead of silently guessing.

## Spec-driven invocation

The most ergonomic long-term shape for real workflows is a TOML verify spec.

Example:

```toml
[program]
entrypoint = "mypkg.workflow:build_program"
params_file = "verification/smoke.program.json"

[environment]
entrypoint = "mypkg.runtime:build_environment"
params_file = "verification/smoke.environment.json"

[runtime]
invocation_name = "real_smoke"
runs_dir = ".runs"
```

This keeps Python factories as the source of truth for typed object
construction while giving the CLI a clean, reproducible invocation surface.

`runs_dir` and any `params_file` entries are resolved relative to the TOML spec
file itself, not the current shell working directory.

## Exit codes

| Code | Meaning |
| --- | --- |
| `0` | Analysis passed, runtime succeeded, and all property checks passed |
| `1` | One or more verification layers failed |

## When to use verify instead of check

Use `check` when you want:

- fast structural feedback
- no runtime execution
- no run bundle

Use `verify` when you want:

- a real execution
- persisted artifacts
- invariant runtime proof
- property-check execution

Most meaningful workflow changes should end with `verify`, not just `check`.
