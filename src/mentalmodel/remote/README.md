# mentalmodel.remote

Remote runs data-plane package for contracts, sink seams, and the repo-owned
hosted-service seam, plus the local stack helpers used for development.

## Purpose

This package defines the typed contracts and the first transport/storage
primitives for remote-compatible run handling:

- canonical run manifests
- named artifact descriptors
- completed-run and execution-record sink interfaces
- project/workspace registration
- repo-owned `mentalmodel.toml` project config loading
- remote project records and catalog snapshot publication
- dedicated catalog republish support for already linked projects
- automatic completed-run upload from repo-linked `mentalmodel verify`
- explicit collector-targeted live execution export through
  `LiveIngestionConfig` / `--live-otlp-endpoint`
- automatic completed-run upload from programmatic `run_managed(...)`
  execution in linked repos
- project-scoped catalog provider shape
- canonical run bundle upload payloads
- a deterministic file-backed remote ingest store
- a Postgres-backed manifest index
- an S3-compatible object-store backend for artifact blobs
- a remote-backed run repository that materializes bundles into a local cache
- a CLI/API sync path for uploading persisted local runs
- workspace TOML load/write helpers for one shared local stack
- localhost bootstrap and doctor flows for the collector-first local stack

The current hosted storage path stores:

- manifest/index rows in Postgres
- artifact bytes in S3-compatible object storage
- a local read cache under `.runs` only as a materialization layer for the
  existing inspection helpers

The current live producer path is OTLP-native:

- spans export as OTLP traces
- metrics export as OTLP metrics
- semantic records export as OTLP logs
- the producer writes to a durable local outbox and drains to an OTel Collector

The older file-backed store remains in-tree as a deterministic fallback for
tests and local stack scenarios.

The collector-first local stack generated by `mentalmodel remote write-demo`
includes:

- Postgres for metadata and indexed completed-run reads
- MinIO for artifact blobs
- Redpanda as the Kafka-compatible event backbone
- ClickHouse provisioned for the unified query model that follows
- OTel Collector configured for OTLP receive, batching, retry, and Kafka routing

The current hosted service still contains a legacy live-session store used by
the dashboard's transitional live read path. That path is not the blessed
producer contract going forward.

The collector/Kafka/ClickHouse stack is therefore provisioned ahead of the
unified hosted query path. In this phase, it is an ingestion backbone and local
operator recipe, not yet the dashboard's authoritative live read model.

Remote delivery health is still surfaced explicitly:

- outbound remote operations use deterministic retry/backoff with explicit
  failure categorization
- project link, catalog publish, and completed run upload traffic
  all record remote operation events
- producer-side verification results surface both
  `runtime.live_execution_delivery` and `runtime.completed_run_upload`
- the dashboard and API can inspect recent remote operation events and
  summarize project/run delivery health
- `mentalmodel remote sync` is the recovery and backfill tool, not the normal
  operator path

## Boundaries

- Keep these types transport-neutral.
- Keep dashboard JSON shapes and storage-specific details at the edges.
- Treat local `.runs` and remote completed-run storage as two backends for the same
  manifest/artifact model.
- Keep the read cache as an implementation detail, not the source of truth.
- Keep the file-backed store deterministic so it can serve as a testable
  fallback alongside the real backend.
- Do not couple the producer live path to the hosted API; the collector is the
  live ingress front door.

## Current flow

1. Local verification or managed execution persists a run in
   `.runs/<graph_id>/<run_id>/...`.
2. When live export is enabled, the runtime appends OTLP-compatible logs,
   traces, and metrics into the durable producer outbox and the async exporter
   drains them to an OTel Collector.
3. `build_run_bundle_upload(...)` resolves a completed run and emits a canonical
   `RunManifest` plus base64-encoded artifact bodies.
4. `POST /api/remote/runs` accepts the completed upload payload and validates it.
5. `RemoteRunStore.ingest(...)` uploads artifact blobs to S3-compatible object
   storage, stores the indexed manifest/read-model row in Postgres, and
   materializes the bundle into a local cache.
6. The dashboard read APIs inspect the indexed remote run through the same
   historical run surfaces by materializing bundles from the remote backend into
   the cache on demand.
7. `FileRemoteRunStore.ingest(...)` remains available as a deterministic fallback.
8. `mentalmodel remote write-demo` generates a workspace registry plus helper
   scripts for one local multi-project collector-first development stack.
9. `mentalmodel remote link` reads repo-owned `mentalmodel.toml`, resolves the
   configured catalog provider, and publishes a remote project record plus the
   current catalog snapshot.
10. `mentalmodel remote publish-catalog` republishes the hosted dashboard
   contract without relinking the whole project.
11. After local materialization succeeds, `mentalmodel verify` uploads the
    completed bundle automatically through `POST /api/remote/runs`.
12. `run_managed(...)` follows the same local-bundle plus optional
    collector-live/completed-upload path without requiring CLI verification
    entrypoints.
13. `mentalmodel remote sync` remains the manual recovery and backfill path
    when automatic upload fails or older local bundles need to be published.
