The runtime has one job: execute the authored semantic graph without losing the
meaning of that graph.

That is why `mentalmodel` does not jump straight from Python objects to
execution. It goes through a compiled runtime model:

1. authored primitives
2. lowered IR graph
3. compiled execution plan
4. async execution
5. semantic records and persisted run artifacts

## The compilation pipeline

Before a workflow runs, `mentalmodel` compiles it in two stages.

### 1. Lower authored primitives into IR

`lower_program(workflow)` traverses the authored model and emits an `IRGraph`.

Core primitives lower directly.
Plugin primitives lower through the plugin registry.

The resulting graph is the shared substrate for:

- analysis
- docs
- Mermaid
- execution-plan compilation

### 2. Compile IR into executable plan nodes

`compile_program(workflow)` then turns the lowered graph into a typed
`ExecutionPlan`.

Each executable node becomes one of:

- `CompiledActorNode`
- `CompiledEffectNode`
- `CompiledJoinNode`
- `CompiledInvariantNode`
- `CompiledPluginNode`

This is an important boundary. The runtime does not execute raw authoring
objects directly.

## AsyncExecutor

`AsyncExecutor` runs a compiled program to completion.

```python
import asyncio
from mentalmodel.runtime.executor import AsyncExecutor

executor = AsyncExecutor(max_concurrency=8)
result = asyncio.run(executor.run(workflow))
```

The returned `ExecutionResult` includes:

- `run_id`
- `graph`
- `outputs`
- `records`
- `state`
- `spans`
- `trace_summary`

## How scheduling works

The executor is dependency-driven.

- nodes with no dependencies start ready
- when a node finishes, the executor checks which downstream nodes now have all
  dependencies satisfied
- those nodes become ready
- the executor launches up to `max_concurrency` ready nodes at a time

The ready queue is sorted deterministically, which helps with stable testing and
replay. But the most important thing to remember is:

**causality comes from dependency edges, not from child order in the source.**

## ExecutionContext

Every handler receives an `ExecutionContext`.

This context carries:

- the run id
- the graph
- the recorder
- tracing and metric emitters
- the shared state store
- the accumulated output map
- node-local metadata such as `node_id`, `node_kind`, and `runtime_context`

That makes handlers execution-aware without making them responsible for runtime
orchestration.

## Semantic execution records

The most important runtime output is the semantic event stream.

Common events include:

| Event type | Meaning |
| --- | --- |
| `node.started` | The executor began a node |
| `node.inputs_resolved` | Bound inputs were materialized |
| `node.succeeded` | The node completed successfully |
| `node.failed` | The node raised |
| `state.read` | An actor read prior state |
| `state.transition` | An actor wrote next state |
| `effect.invoked` | An effect began |
| `effect.completed` | An effect returned |
| `join.resolved` | A join saw all inputs and resolved |
| `invariant.checked` | An invariant executed |

These records are the package's primary debugging substrate.

OTel spans are useful, but they are secondary to the semantic records.

## `.runs` bundles

Verification and debugging flows persist a run bundle under:

```text
.runs/<graph_id>/<run_id>/
```

Important files include:

- `summary.json`
- `verification.json`
- `records.jsonl`
- `outputs.json`
- `state.json`
- `otel-spans.jsonl` when local span mirroring is enabled

This is the main local debugging surface. A good mental model is:

- traces are for observability integration
- `.runs` is for deterministic local understanding

## Determinism

The runtime tries to remain deterministic where practical.

Today that includes:

- stable node ids
- stable edge ids
- sorted dependency tuples
- sorted ready queue behavior
- stable `.runs/<graph_id>/<run_id>/` layout

That determinism is what makes:

- replay
- run diffing
- property verification
- agent debugging

much more useful.

## Current boundaries and limitations

A few things to understand clearly:

- the runtime is DAG-oriented today
- there is not yet a native loop primitive
- execution is one run of one compiled workflow
- `.runs` and replay are therefore run-scoped, not iteration-scoped yet

That is an active area of planned package work, not an accidental omission.

## The most useful commands to pair with the runtime

```bash
mentalmodel verify --entrypoint module:function
mentalmodel runs latest --graph-id <graph_id>
mentalmodel runs inputs --graph-id <graph_id> --node-id <node_id>
mentalmodel runs outputs --graph-id <graph_id> --node-id <node_id>
mentalmodel runs trace --graph-id <graph_id> --node-id <node_id>
mentalmodel replay --graph-id <graph_id>
```

If you are trying to understand what happened, start there before reaching for
an external debugger.
