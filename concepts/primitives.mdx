---
title: "Primitives"
description: "The semantic building blocks of a mentalmodel program: what each primitive means and how it is meant to be used."
---

Every `mentalmodel` program is authored from a small set of primitives. The key
idea is that each primitive has one semantic job.

That is what makes the resulting program readable:

- `Actor` means state
- `Effect` means side effect or impure boundary
- `Invariant` means correctness condition
- `Ref` means dependency edge

These primitives lower into a shared IR graph, and that same IR powers:

- analysis
- diagrams
- generated docs
- runtime execution
- replay and `.runs`

## A quick mental model

Think of the primitives in three groups:

### Structural

- `Workflow`
- `Block`
- `Use`
- `Parallel`
- `RuntimeContext`

These group other nodes and shape the readable structure of the program.

### Executable

- `Actor`
- `Effect`
- `Join`
- `Invariant`

These become runtime-executable nodes.

### Linking

- `Ref`
- `BlockRef`

This is how data dependencies are declared.

## Workflow

`Workflow` is the top-level semantic container for a program.

```python
from mentalmodel import Workflow

program = Workflow(
    name="hello_world",
    description="Minimal workflow.",
    children=[...],
)
```

Use it to:

- name the program
- group its top-level nodes
- define the graph id used in `.runs`

`Workflow` is a container primitive. It lowers to a `workflow` node plus
`contains` edges to its children.

## Actor

`Actor` is the primitive for stateful computation.

```python
from mentalmodel import Actor, Ref

Actor(
    "cursor",
    handler=CursorHandler(),
    inputs=[Ref("source")],
)
```

Use `Actor` when:

- the node reads prior state
- the node writes next state
- the output depends on accumulated or durable state

At runtime, an actor handler receives:

- resolved typed inputs
- previous state, if any
- an `ExecutionContext`

and returns an `ActorResult(output=..., next_state=...)`.

## Effect

`Effect` is the primitive for explicit impure boundaries.

```python
from mentalmodel import Effect, Ref

Effect(
    "fetch_profile",
    handler=FetchProfile(),
    inputs=[Ref("user_id_source")],
)
```

Use `Effect` for:

- network calls
- filesystem access
- database operations
- model inference calls
- any observable external operation

`Effect` is intentionally explicit. If something has side effects, it should not
be disguised as an `Actor`.

## Invariant

`Invariant` attaches a named runtime check to explicit values.

```python
from mentalmodel import Invariant, Ref

Invariant(
    "profile_not_empty",
    checker=ProfileNotEmpty(),
    inputs=[Ref("fetch_profile")],
)
```

Use it for:

- value-range checks
- key-set alignment
- causal ordering
- monotonicity constraints
- domain-specific correctness properties

At runtime, invariants are first-class nodes. They execute and emit records.
Failed `severity="error"` invariants fail the run; failed `severity="warning"`
invariants are surfaced but non-fatal.

## Parallel

`Parallel` is a structural grouping for fanout work.

```python
from mentalmodel import Parallel

Parallel(
    "reward_fanout",
    children=[
        ...,
        ...,
    ],
)
```

Use it to make concurrency readable in authored code. `Parallel` does not
create new dependency semantics by itself; the actual dataflow still comes from
`Ref`.

That distinction matters:

- `Parallel` improves human legibility
- `Ref` is what makes nodes runnable

## Join

`Join` is the explicit synchronization and merge point after fanout.

```python
from mentalmodel import Join, Ref

Join(
    "rollout_join",
    inputs=[Ref("reward_a"), Ref("reward_b")],
    reducer=RolloutJoinReducer(),
)
```

Use it when:

- multiple upstream values must all be present before continuing
- you want a reducer to combine them into one typed output
- you want the merge point to be visible in the graph and records

Without a reducer, a `Join` passes the resolved input mapping through directly.

## Ref

`Ref` declares a dependency on another node's output.

```python
from mentalmodel import Ref

Ref("sample_policy")
Ref("sample_policy", port="default")
```

`Ref` is how non-tree edges are expressed. It is the authored signal that:

- this node depends on that node
- the dependency is dataflow, not just containment

`Ref` targets do not need to appear earlier in the `children` list. They just
need to resolve to real node ids after lowering.

## Block and Use

`Block` defines a reusable semantic fragment. `Use` instantiates that fragment
under a namespace.

```python
from mentalmodel import Block, BlockInput, BlockOutput, BlockRef, Use
```

Use them when:

- a fragment is reused more than once
- manual prefix rewriting would otherwise leak into authored source
- you want logical block inputs and outputs instead of wiring through raw
  internal node ids

See [Reusable blocks](/guides/reusable-blocks) for the full model.

## RuntimeContext

`RuntimeContext` is the main built-in plugin primitive today. It groups a
subtree under a runtime boundary such as `local` or `sandbox`.

```python
from mentalmodel import RuntimeContext

RuntimeContext(
    name="remote_sampling",
    runtime="sandbox",
    children=[...],
)
```

Use it when you want runtime placement to be explicit in:

- analysis
- docs
- Mermaid
- records
- span attributes

It is not a core primitive implemented directly in `mentalmodel.core`; it is a
plugin-based extension that proves the extension model is part of the real
package, not an afterthought.

## What is actually guaranteed today

Some important clarifications:

- There is a real `LowersToIR` protocol in the codebase, but most users will
  interact with primitives and plugins rather than the protocol directly.
- `Parallel` is a grouping signal, not a magic scheduler override.
- Child order is useful for readability, but true execution order comes from
  dependency edges and runtime readiness.
- `StepLoop` is the current native iteration primitive. It is intentionally
  narrower than a general `while` loop: sequential, block-based, and explicit
  about loop-carried state.

## How to choose the right primitive

Ask these questions:

| Question | Primitive |
| --- | --- |
| Does this node own state? | `Actor` |
| Does this node perform I/O or any side effect? | `Effect` |
| Do I need to assert correctness on concrete values? | `Invariant` |
| Am I fanning out independent work? | `Parallel` |
| Am I synchronizing multiple upstream values? | `Join` |
| Am I declaring a dependency edge? | `Ref` |
| Am I iterating a reusable block over a list of items? | `StepLoop` |
| Am I grouping nodes under a runtime boundary? | `RuntimeContext` |
