---
title: "Metrics"
description: "How built-in runtime metrics and output-derived metrics work in mentalmodel."
---

`mentalmodel` exposes two different metric layers:

1. built-in operational metrics emitted by the runtime
2. output-derived domain metrics attached to `Actor` and `Effect` nodes

Both go through the same metric emitter interface, but they serve different
purposes.

## Built-in operational metrics

These are emitted automatically by the runtime:

| Metric name | Kind | Meaning |
| --- | --- | --- |
| `mentalmodel.run.started` | counter | One run started |
| `mentalmodel.run.completed` | counter | One run completed, tagged by success |
| `mentalmodel.node.executions` | counter | One node execution started |
| `mentalmodel.node.duration_ms` | histogram | One node execution duration |
| `mentalmodel.invariant.failures` | counter | One invariant failed |

These are good for:

- high-level run tracking
- latency trends
- failure counting
- runtime-context and node-kind breakdowns

They are intentionally low-cardinality.

## Output-derived metrics

When a node returns domain data that should become metrics, attach metric
policies to the node.

Today this is supported on:

- `Actor`
- `Effect`

through the `metrics=[...]` parameter.

There are three main ways to do it.

## `infer_output_metrics(...)`

Use this when the output is already a small flat numeric summary.

```python
from mentalmodel import Actor, Ref, infer_output_metrics

Actor(
    "learner_update",
    handler=LearnerUpdate(),
    inputs=[Ref("rollout_join")],
    metrics=[infer_output_metrics(prefix="mentalmodel.demo.learner_update")],
)
```

This works well for outputs such as:

```python
{
    "updated_policy_version": 4,
    "sample_count": 8,
}
```

The inference path is deliberately conservative:

- output must be a flat mapping
- keys must be safe metric names
- values must be numeric
- mappings with too many fields are skipped

This is a guardrail against accidental high-cardinality metric export.

## `extract_output_metrics(...)`

Use this when the output needs aggregation or custom shaping.

```python
from dataclasses import dataclass
from mentalmodel import (
    MetricContext,
    MetricDefinition,
    MetricExtractor,
    MetricKind,
    MetricObservation,
    extract_output_metrics,
)


RewardScores = dict[str, float]


@dataclass(slots=True, frozen=True)
class RewardSummaryMetrics(MetricExtractor[RewardScores]):
    prefix: str

    def extract(
        self,
        output: RewardScores,
        context: MetricContext,
    ) -> tuple[MetricObservation, ...]:
        if not output:
            return tuple()
        values = list(output.values())
        attrs = context.default_attributes()
        return (
            MetricObservation(
                definition=MetricDefinition(
                    name=f"{self.prefix}.mean",
                    kind=MetricKind.HISTOGRAM,
                    description="Mean reward value.",
                ),
                value=sum(values) / len(values),
                attributes=dict(attrs),
            ),
            MetricObservation(
                definition=MetricDefinition(
                    name=f"{self.prefix}.count",
                    kind=MetricKind.HISTOGRAM,
                    description="Number of scored items.",
                ),
                value=len(values),
                attributes=dict(attrs),
            ),
        )
```

Then attach it:

```python
Effect(
    "pangram_reward",
    handler=PangramScorer(),
    inputs=[Ref("sample_policy")],
    metrics=[
        extract_output_metrics(
            RewardSummaryMetrics(prefix="mentalmodel.demo.reward.pangram")
        )
    ],
)
```

This path is what you want for:

- dynamic maps
- aggregated reward summaries
- domain-specific counters
- custom projections that need logic beyond named-field selection

## `project_metric_map(...)` and `project_flat_metric_map(...)`

Use these when the node output already contains stable numeric fields and you
want a small exported subset without writing a custom extractor class.

This is the intended path for provider-native metric maps such as
`optimizer_metrics`, where you want to preserve the full raw map in the output
and `.runs`, but only emit a few stable fields as metrics.

```python
from mentalmodel import (
    Effect,
    MetricFieldProjection,
    MetricMapProjection,
    Ref,
    project_metric_map,
)


Effect(
    "optimizer_update",
    handler=OptimizerUpdate(),
    inputs=[Ref("rollout_join")],
    metrics=[
        project_metric_map(
            MetricMapProjection(
                metric_map_key="optimizer_metrics",
                metric_name_prefix="mentalmodel.demo.optimizer",
                fields=(
                    MetricFieldProjection("loss", "loss"),
                    MetricFieldProjection("approx_kl", "approx_kl"),
                ),
            )
        )
    ],
)
```

For typed outputs that are not already shaped like a nested metric map, use the
flat helper with an accessor:

```python
from mentalmodel import project_flat_metric_map


metrics = [
    project_flat_metric_map(
        prefix="mentalmodel.demo.answer",
        fields=("total_monthly_cost", "success_score"),
        accessor=lambda output: {
            "total_monthly_cost": output["total_monthly_cost"],
            "success_score": output["success_score"],
        },
    )
]
```

Projection helpers are intentionally conservative:

- they only emit explicitly named fields
- missing fields are skipped
- non-numeric values are skipped
- they do not automatically export every key in a provider map

## Important design rule

Do not emit one metric per example or per dynamic id by default.

For example, this output:

```python
{
    "sample-123": 0.81,
    "sample-124": 0.77,
    "sample-125": 0.95,
}
```

should usually be aggregated first with `extract_output_metrics(...)`, not
exported directly as metrics.

That kind of data belongs better in:

- node outputs
- semantic records
- traces
- `.runs`

than in default low-cardinality metrics.

## Metric context

Custom extractors receive a `MetricContext` that carries stable runtime
attributes such as:

- `graph_id`
- `run_id`
- `node_id`
- `node_kind`
- `runtime_context`
- `service_name`

Use `context.default_attributes()` as the base for emitted observations.

## When metrics are actually exported

Metrics are only exported when the configured observability mode creates a real
metric emitter:

- `otlp_http`
- `otlp_grpc`
- `console`

When the mode is `disk` or `disabled`, metrics are not exported externally.

That does **not** make the runtime blind:

- `.runs` still exists
- outputs and records still exist
- verification still works

But it does mean you should not expect a local metrics backend unless you have
configured one.

## OTLP examples

<CodeGroup>

```bash OTLP HTTP
MENTALMODEL_OTEL_MODE=otlp_http \
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318/v1/traces \
python my_program.py
```

```bash OTLP gRPC
MENTALMODEL_OTEL_MODE=otlp_grpc \
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4317 \
python my_program.py
```

```bash Console
MENTALMODEL_OTEL_MODE=console python my_program.py
```

</CodeGroup>

If you are using the self-hosted demo stack, prefer the helper flow described in
[observability/opentelemetry](/observability/opentelemetry) and
[docs/otel-self-hosted](/docs/otel-self-hosted).

## What this should feel like in practice

A good `mentalmodel` metric setup should make you feel like:

- the runtime already gives you useful operational metrics
- node outputs can become domain metrics with a small amount of explicit code
- you do not need to sprinkle raw metric calls throughout handlers

That is the standard the package is aiming for.
