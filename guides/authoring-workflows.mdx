---
title: "Authoring Workflows"
description: "How to structure mentalmodel workflows so they stay readable as they grow."
---

The fastest way to make a `mentalmodel` program unreadable is to treat
`Workflow(children=[...])` as a dumping ground.

The fastest way to keep it readable is to let the top-level workflow express
architecture, not implementation detail.

This guide focuses on that authored shape.

## The basic rule

Use containers to express structure and `Ref` to express dataflow.

- `Workflow`, `Parallel`, `RuntimeContext`, and `Use` provide readable grouping
- `Ref` provides executable dependency edges

If a reader can understand your workflow by looking at the containers and a few
well-named nodes, you are on the right path.

## Start with a program factory

Always build workflows in a function that returns a `Workflow`.

```python
from mentalmodel import Workflow

def build_program() -> Workflow:
    return Workflow(
        name="hello_world",
        description="Minimal example workflow.",
        children=[...],
    )
```

This keeps the authored model:

- parameterizable
- testable
- easy to pass to CLI entrypoints
- easy to reuse in demos, verification, and optimization flows

## Use containers for semantic grouping

A good top-level workflow usually reads like a list of major subsystems.

```python
Workflow(
    name="async_rl_demo",
    children=[
        RuntimeContext(
            name="local_control_plane",
            runtime="local",
            children=[...],
        ),
        RuntimeContext(
            name="remote_sampling",
            runtime="sandbox",
            children=[...],
        ),
    ],
)
```

That is better than a flat list of twenty nodes with no visual structure.

## Use names that describe roles, not implementations

Good names:

- `batch_source`
- `sample_policy`
- `reward_join`
- `staleness_invariant`
- `learner_update`

Bad names:

- `handler1`
- `step2`
- `foo`
- `reward_join_v2`

Node names become:

- IR ids
- `.runs` keys
- replay filters
- span names
- diff targets

So naming is not cosmetic.

## `Ref` is where dataflow becomes real

Containers do not create data dependencies by themselves. `Ref` does.

```python
Effect(
    "sample_policy",
    handler=PolicySampler(),
    inputs=[Ref("batch_source"), Ref("policy_snapshot")],
)
```

That is the line that tells the runtime:

- which outputs this node consumes
- when it becomes runnable
- what should be persisted in `node.inputs_resolved`

Treat `Ref` edges as part of the architecture, not as minor wiring.

## Prefer explicit fanout and join

When work is conceptually parallel, say so in the source.

```python
Parallel(
    "reward_fanout",
    children=[
        Effect("pangram_reward", handler=PangramScorer(), inputs=[Ref("sample_policy")]),
        Effect("quality_reward", handler=QualityScorer(), inputs=[Ref("sample_policy")]),
    ],
)

Join(
    "reward_join",
    inputs=[Ref("pangram_reward"), Ref("quality_reward")],
    reducer=RewardJoinReducer(),
)
```

This gives you:

- a more readable graph
- a more readable Mermaid view
- a more readable replay trace

## Keep handlers domain-focused

Handlers should contain domain logic, not graph plumbing.

If you find yourself writing handler code that knows about prefixed node ids or
container names, that is usually a sign the authored structure is getting too
stringly.

For reusable fragments, prefer `Block` plus `Use` so namespacing and block
input binding stay in the authoring layer instead of leaking into handlers.

## A practical structure for medium-sized workflows

For current `mentalmodel`, a good pattern is:

1. top-level `Workflow`
2. a small number of semantically meaningful container blocks
3. executable leaf nodes inside each block
4. explicit joins and invariants near the points where meaning is established

Example:

```python
Workflow(
    name="async_rl_demo",
    children=[
        RuntimeContext(
            name="local_control_plane",
            runtime="local",
            children=[
                Actor("policy_snapshot", handler=...),
                Actor("batch_source", handler=...),
                RuntimeContext(
                    name="remote_sampling",
                    runtime="sandbox",
                    children=[
                        Effect("sample_policy", handler=..., inputs=[...]),
                    ],
                ),
                Parallel(
                    name="reward_fanout",
                    children=[
                        Effect("pangram_reward", handler=..., inputs=[Ref("sample_policy")]),
                        Effect("quality_reward", handler=..., inputs=[Ref("sample_policy")]),
                    ],
                ),
                Join("rollout_join", inputs=[...], reducer=...),
                Invariant("staleness_invariant", checker=..., inputs=[Ref("rollout_join")]),
                Actor("learner_update", handler=..., inputs=[Ref("rollout_join")]),
            ],
        ),
    ],
)
```

The top-level shape tells the reader roughly what the program does before they
need to read any handler implementation.

## What not to assume about child order

Child order is useful for readability and produces deterministic lowering, but
it is not the real execution contract.

The runtime executes nodes based on dependency readiness. If two nodes become
ready at the same time, the executor uses deterministic ordering internally,
but the true causal structure comes from `Ref` edges.

So:

- use child order to make the authored source pleasant to read
- use `Ref` to make execution semantics explicit

## Use the tooling while authoring

The most useful authoring loop is:

```bash
mentalmodel check --entrypoint demo:build_program
mentalmodel graph --entrypoint demo:build_program
mentalmodel docs --entrypoint demo:build_program
mentalmodel verify --entrypoint demo:build_program
```

That gives you:

- structural validation
- topology visibility
- generated inventories
- real runtime proof

If the graph or docs become hard to read, the workflow is probably too hard to
read too.

## Current limitation

The package now has first-class reusable block instantiation via `Block` and
`Use`, but it does not yet have native loop control flow. Larger iterative
workflows still need extra authoring discipline until `StepLoop` lands.
