title: "Quick start"
description: "Install mentalmodel, build a small workflow, and inspect its runtime artifacts."
---

This page intentionally starts with a tiny workflow, not the full async RL
demo. The goal is to learn the model first:

1. author a small semantic graph
2. run static analysis
3. execute it
4. inspect the resulting `.runs` bundle

Once that makes sense, the bigger demos will feel much more legible.

## Prerequisites

- Python 3.11 or newer
- `uv` recommended, though `pip` also works

<Steps>
  <Step title="Install mentalmodel">
    <CodeGroup>

    ```bash uv
    uv add mentalmodel
    ```

    ```bash pip
    pip install mentalmodel
    ```

    </CodeGroup>

    Verify the CLI is available:

    ```bash
    mentalmodel --help
    ```
  </Step>

  <Step title="Write a tiny workflow">
    Create `demo.py`:

    ```python
    from __future__ import annotations

    from dataclasses import dataclass
    from typing import TypedDict

    from mentalmodel import Actor, Effect, Invariant, Ref, Workflow
    from mentalmodel.core import ActorHandler, EffectHandler, InvariantChecker
    from mentalmodel.core.models import ActorResult, InvariantResult
    from mentalmodel.runtime.context import ExecutionContext


    class SourceInputs(TypedDict):
        pass


    class GreetingState(TypedDict):
        count: int


    class GreetingOutput(TypedDict):
        greeting: str
        count: int


    class UppercaseInputs(TypedDict):
        greeting_source: GreetingOutput


    class UppercaseOutput(TypedDict):
        message: str


    class MessageInvariantInputs(TypedDict):
        uppercase_message: UppercaseOutput


    @dataclass(slots=True)
    class GreetingSource(ActorHandler[SourceInputs, GreetingState, GreetingOutput]):
        async def handle(
            self,
            inputs: SourceInputs,
            state: GreetingState | None,
            ctx: ExecutionContext,
        ) -> ActorResult[GreetingOutput, GreetingState]:
            del inputs, ctx
            next_count = 1 if state is None else state["count"] + 1
            output: GreetingOutput = {
                "greeting": "hello mentalmodel",
                "count": next_count,
            }
            next_state: GreetingState = {"count": next_count}
            return ActorResult(output=output, next_state=next_state)


    @dataclass(slots=True)
    class UppercaseMessage(EffectHandler[UppercaseInputs, UppercaseOutput]):
        async def invoke(
            self,
            inputs: UppercaseInputs,
            ctx: ExecutionContext,
        ) -> UppercaseOutput:
            del ctx
            return {"message": inputs["greeting_source"]["greeting"].upper()}


    @dataclass(slots=True)
    class MessageNotEmpty(InvariantChecker[MessageInvariantInputs, str]):
        async def check(
            self,
            inputs: MessageInvariantInputs,
            ctx: ExecutionContext,
        ) -> InvariantResult[str]:
            del ctx
            message = inputs["uppercase_message"]["message"]
            return InvariantResult(
                passed=bool(message.strip()),
                details={"message": message},
            )


    def build_program() -> Workflow:
        return Workflow(
            name="hello_world",
            description="A minimal workflow with one actor, one effect, and one invariant.",
            children=[
                Actor("greeting_source", handler=GreetingSource()),
                Effect(
                    "uppercase_message",
                    handler=UppercaseMessage(),
                    inputs=[Ref("greeting_source")],
                ),
                Invariant(
                    "message_not_empty",
                    checker=MessageNotEmpty(),
                    inputs=[Ref("uppercase_message")],
                ),
            ],
        )
    ```

    This example shows the basic shape of the model:

    - `Actor` owns state
    - `Effect` performs a transformation at an explicit boundary
    - `Invariant` checks a property of a concrete output
    - `Ref` makes the data dependencies explicit
  </Step>

  <Step title="Run static analysis">
    ```bash
    mentalmodel check --entrypoint demo:build_program
    ```

    On success you should see a short summary and no findings.

    Example:

    ```text
    mentalmodel check summary
    Graph       Nodes  Edges  Errors  Warnings
    hello_world 4      3      0       1

    Findings
    Severity  Code                     Node               Message
    warning   missing-runtime-context  greeting_source    Node does not inherit or declare a runtime context.
    ```

    How to read that:

    - `Graph` is the workflow name and the future `.runs` graph id
    - `Nodes` and `Edges` tell you how large the lowered graph is
    - `Errors` block execution confidence; `Warnings` are usually design-quality signals
    - the `missing-runtime-context` warning is common in tiny examples and means
      you have not yet declared where that node runs

    Today, `check` verifies structural and semantic properties such as:

    - duplicate node ids
    - malformed graph edges
    - missing workflow roots
    - missing runtime-context metadata
    - missing plugin provenance

    It does not try to prove arbitrary business correctness from your code.

    Useful next step:

    - if you see `duplicate-node-id` or edge errors, fix the authored graph first
    - if you only see warnings, you can usually continue to `verify`
  </Step>

  <Step title="Execute and persist a run bundle">
    ```bash
    mentalmodel verify --entrypoint demo:build_program
    ```

    `verify` performs:

    1. static analysis
    2. runtime execution
    3. property-check execution when the module defines any

    It also writes a run bundle under:

    ```text
    .runs/hello_world/<run_id>/
    ```

    Example summary:

    ```text
    mentalmodel verify summary
    Graph       Static Errors  Warnings  Runtime  Property Checks
    hello_world 0              1         pass     0

    Runtime Verification
    Success  Records  Outputs  State Entries  Run Artifacts
    yes      10       3        1              .runs/hello_world/run-abc123
    ```

    How to read that:

    - `Runtime pass` means the workflow actually executed successfully
    - `Records` is how many semantic events were emitted
    - `Outputs` is how many nodes produced persisted outputs
    - `State Entries` matters for actors; here it shows the actor state was recorded
    - `Run Artifacts` is the directory you will inspect next
  </Step>

  <Step title="Inspect what actually happened">
    These commands are the fastest way to build intuition:

    ```bash
    mentalmodel runs latest --graph-id hello_world
    mentalmodel runs outputs --graph-id hello_world --node-id uppercase_message
    mentalmodel runs inputs --graph-id hello_world --node-id message_not_empty
    mentalmodel runs trace --graph-id hello_world --node-id message_not_empty
    mentalmodel replay --graph-id hello_world
    ```

    This is the core `mentalmodel` loop:

    - author the workflow
    - verify it
    - inspect `.runs`
    - refine the graph

    A practical interpretation flow:

    - `runs latest` tells you which run bundle to inspect
    - `runs outputs` answers “what did this node produce?”
    - `runs inputs` answers “what exact payload did this node see?”
    - `runs trace` answers “what happened around this node?”
    - `replay` answers “what happened in the whole run, in order?”
  </Step>

  <Step title="Generate graph and docs">
    ```bash
    mentalmodel graph --entrypoint demo:build_program
    mentalmodel docs --entrypoint demo:build_program --output-dir docs/hello_world
    ```

    `graph` renders a Mermaid topology view.

    `docs` generates IR-derived markdown artifacts, including:

    - topology
    - node inventory
    - invariant catalog
    - runtime-context summary

    These are most useful when:

    - the workflow is getting large enough that you want a high-level review
    - you want to explain the program to another human
    - you want to compare authored structure against runtime behavior
  </Step>
</Steps>

## When to move to a bigger example

Once the tiny workflow above feels clear, use the packaged demos:

```bash
mentalmodel demo async-rl
mentalmodel demo agent-tool-use
mentalmodel demo autoresearch-sorting
```

Those examples show:

- runtime contexts
- fanout and joins
- output-derived metrics
- runtime-executable plugins
- replay and run diffing
