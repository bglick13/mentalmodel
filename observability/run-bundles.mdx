Run bundles are the local debugging backbone of `mentalmodel`.

Whenever you run verification, the package writes a structured bundle under:

```text
.runs/<graph_id>/<run_id>/
```

If you only remember one thing from this page, remember this:

**start with the semantic run bundle before jumping to raw OTel data.**

## What a bundle contains

Typical contents:

```text
.runs/
└── <graph_id>/
    └── <run_id>/
        ├── summary.json
        ├── verification.json
        ├── records.jsonl
        ├── outputs.json
        ├── state.json
        └── otel-spans.jsonl
```

## The files that matter most

### `summary.json`

High-level metadata for the run:

- schema version
- graph id
- run id
- success flag
- node and edge counts
- record/output/state counts
- resolved tracing summary

Use this first when you want to know:

- what run you are looking at
- whether it succeeded
- what tracing mode was active

### `verification.json`

The combined verification result written by `mentalmodel verify`.

Use this first when a run failed. It gives you the fastest structured answer to:

- did analysis fail?
- did runtime execution fail?
- did any warning-severity invariants fail while the run still succeeded?
- did any property checks fail?

### `records.jsonl`

This is the main runtime ledger.

Each line is a semantic execution record. This is where you go to answer:

- what ran?
- in what order?
- with what inputs?
- where did it fail?

Common event types include:

- `node.started`
- `node.inputs_resolved`
- `node.succeeded`
- `node.failed`
- `state.read`
- `state.transition`
- `effect.invoked`
- `effect.completed`
- `join.resolved`
- `invariant.checked`

### `outputs.json`

Serialized node outputs keyed by node id.

Use this when you want the actual materialized output of one node, not just the
fact that it ran.

### `state.json`

Final actor state values.

Use this to debug stateful nodes and state transitions.

### `otel-spans.jsonl`

A local mirror of spans when span mirroring is enabled.

This is useful for:

- precise timing
- trace correlation
- exported-span debugging

But it is usually **not** where you should start.

## The best inspection workflow

For most bugs, this sequence works well:

1. `mentalmodel runs latest --graph-id <graph_id>`
2. `mentalmodel runs show --graph-id <graph_id>`
3. `mentalmodel runs trace --graph-id <graph_id> --node-id <node_id>`
4. `mentalmodel runs inputs --graph-id <graph_id> --node-id <node_id>`
5. `mentalmodel runs outputs --graph-id <graph_id> --node-id <node_id>`
6. `mentalmodel replay --graph-id <graph_id>`

That gives you:

- the run you care about
- the node lifecycle
- the exact bound inputs
- the exact produced output
- the broader run timeline

## CLI commands that matter

### List runs

```bash
mentalmodel runs list
mentalmodel runs list --graph-id async_rl_demo
```

### Inspect the latest run

```bash
mentalmodel runs latest --graph-id async_rl_demo
mentalmodel runs show --graph-id async_rl_demo
```

### Inspect one node

```bash
mentalmodel runs inputs --graph-id async_rl_demo --node-id staleness_invariant
mentalmodel runs outputs --graph-id async_rl_demo --node-id rollout_join
mentalmodel runs trace --graph-id async_rl_demo --node-id staleness_invariant
mentalmodel runs records --graph-id async_rl_demo --node-id staleness_invariant
```

### Compare two runs

```bash
mentalmodel runs diff \
  --graph-id async_rl_demo \
  --run-a <run_a_id> \
  --run-b <run_b_id>
```

Narrowing by node or invariant is often the fastest way to localize a
regression.

## Schema versioning

Run bundles are versioned.

`summary.json` carries `schema_version`, and the CLI can repair older bundles
when fields are missing:

```bash
mentalmodel runs repair --dry-run
mentalmodel runs repair
```

This matters because `.runs` is treated as a durable product surface, not a
throwaway debug cache.

## Relationship to OpenTelemetry

The package has two observability layers:

- semantic records and run bundles
- exported traces and metrics

The local bundle is the most complete source for semantic debugging.
OTel is the best layer for:

- backend integration
- dashboards
- cross-service correlation
- live trace exploration

They are complementary, not competing.

## A useful mental model

Use `.runs` when you want to understand:

- structure
- causality
- concrete inputs and outputs
- invariant behavior
- run-to-run differences

Use OTel when you want to understand:

- timing
- distributed behavior
- exported telemetry views
